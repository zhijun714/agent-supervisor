# 验收闸设计（Acceptance Gates）

## 为什么需要多道闸

AI agent 有系统性失败模式——走捷径、幻觉、"自我确认偏差"（报"已验证"却跳过最易翻车的子场景）。2026 年 AI 生成代码占 PR 近 28% 且漏洞更多，瓶颈从"写"转到"审"。单靠 Dev 自述"完成"不可信，故设多道独立闸。

## 设计原则

1. **独立验证 > 自我确认**：每道闸尽量由"非作者"或"工具"把关。
2. **分层防御**：每道闸对应一种具体失败模式，漏网由下一道接住。
3. **概率 + 确定性互补**：LLM 审查（语义/概率）+ 静态工具（穷举/确定）并用。
4. **硬门控优先**：能做成"可验证产物/工具"的，不靠提示词软约束。
5. **可观测、可介入**：每道闸结果在协作记录里可见，人随时接管。

## 闸的分层（按任务生命周期）

| 闸 | 谁把关 | 防的失败模式 | 性质 |
|---|---|---|---|
| ① 验收测试先行 | PA 派任务定义可验证的"完成" | 模糊"做完了"、跑偏 | 前置契约 |
| ② Dev 真实证据 | 报告须含真实命令+输出，禁"读代码确认" | 没验证就报完成 | 自证（硬要求） |
| ③ /code-review | Dev 完成时跑，PA 验收时核 | correctness bug、删 null 检查没改调用方 | 独立工具（概率） |
| ④ jscpd 重复率 | Dev 报告附（提示不卡闸） | 复制粘贴、重复实现 | 静态工具（确定） |
| ⑤ 接口质量(codebase-design) | PA 用深模块词汇验 | 浅模块、过宽接口 | 设计审查 |
| ⑥ scope-creep 驳回 | PA 验收 | 夹带重构、未要求防御/抽象 | 设计审查 |
| ⑦ PA 真验收 | PA 独立运行、枚举子场景 | 自我确认偏差（只跑 happy path） | 独立复核 |
| ⑧ QA 对抗性测试 | 独立 QA 进程，从不读实现 | 实现带偏的盲点、边界/回归 | 进程级独立 |

## 思路要点

- 每道闸 = 一种失败模式的解药，不是堆流程：①治模糊、②治不验证、③④治 bug/重复、⑤⑥治坏设计、⑦⑧治自我确认偏差。
- 独立性两种来源：工具（/code-review、jscpd 无作者偏见）+ 进程隔离（QA 独立进程、没看过实现才能真对抗——即"三个独立进程"的核心价值）。
- 软硬结合：硬的（②真实输出、③工具产物、④静态指标）尽量硬；软的（⑤⑥设计判断）靠 PA + codebase-design 词汇 + 提示词纪律。
- 分档：日常 in-session /code-review；重要/合并前 /code-review ultra（云端多 agent）。琐碎改动跳过重闸，避免仪式过度。
- 不卡死的提示闸：jscpd 作提示而非硬闸——确定性指标有价值，但不该让重复率数字阻塞交付，由 PA 判断。
